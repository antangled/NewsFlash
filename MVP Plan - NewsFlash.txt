Here’s a detailed MVP instruction plan that keeps the idea tight and avoids feature creep.
Core MVP goal
Build a product that:
* watches a curated set of high-signal tech/startup Twitter/X accounts
* extracts the most important posts from the last 24 hours
* ranks them by likely impact
* turns them into very short summaries
* pushes them in a low-effort, impossible-to-ignore format
The key is not “best tech news app.”
The key is:
“I don’t need to open anything. The important tech news just appears.”

5. Define what counts as “important”
You already identified the hard part:
users don’t know what to prioritize.
So your product must act like an editor.
Create a simple impact score
For MVP, score each candidate post on 4 factors:
A. Source importance
How important is the poster?
Example scale:
* 5 = category-defining leader/company
* 4 = major leader/company
* 3 = respected insider or large startup
* 2 = relevant but secondary voice
* 1 = everyone else
B. Event importance
What kind of thing is being said?
Higher score for:
* launch
* partnership
* funding
* acquisition
* policy/regulation
* major hiring signal
* product release
* infrastructure/capability jump
* pricing/business model change
* public stance that shifts industry narrative
Lower score for:
* commentary
* jokes
* general takes
* motivational posts
C. Engagement velocity
How fast is it spreading?
Use:
* likes
* reposts
* replies
* maybe views if available
Not raw totals only.
Better:
* engagement relative to account size
* engagement within first X hours
D. Cross-source confirmation
If multiple high-signal accounts talk about the same thing, it matters more.
________________


6. Use a dead-simple ranking formula
Do not overbuild ML first.
Use a heuristic score like:
Total Score = (Source Weight × 0.35) + (Event Type Weight × 0.35) + (Engagement Velocity × 0.2) + (Multi-source Confirmation × 0.1)
Then rank top items.
Add one important rule:
Cluster duplicates.
If 8 people are talking about the same launch, do not show 8 headlines.
Show:
* one main headline
* 1–2 source references behind it
That makes the product feel editorial instead of noisy.

10. Suggested MVP system architecture
Keep this stupid simple.
Pipeline
Step 1: Fetch posts
Every few hours or once daily:
* pull posts from curated accounts
* limit to posts from last 24 hours
Step 2: Filter candidate posts
Remove:
* pure replies
* low-engagement noise
* obvious memes/jokes unless massively important
* duplicates
* non-news chatter
Step 3: Score posts
Apply heuristic importance score.
Step 4: Cluster related posts
Group together posts about the same event/topic.
Step 5: Summarize clusters
Use LLM to generate:
* one-line headline
* one-line supporting explanation
Step 6: Store final top stories
Save top 3–7 for the day.
Step 7: Deliver
Extension reads today’s stories from backend and displays them.

You need to think about this immediately.
MVP instruction

If Twitter/X API is painful or expensive, your contingency options are:

13. Build manual before automated
Before writing much code, simulate the product manually for 3–5 days.
Manual test process
Each day:
* review posts from 30–50 target accounts
* pick top 5 stories
* write one-line summaries
* mock them in a bottom-bar design or notion page
* show to 5–10 target users
Ask:
* did this save you time?
* did you learn anything meaningful?
* was the ranking good?
* was it too noisy?
* would you install this?
* would you keep it on?
This manual phase will teach you:
* what actually counts as signal
* how many headlines is too many
* what summary length works
* whether users prefer founders, AI, startup funding, or product launches
This is way more valuable than prematurely coding everything.

14. User testing questions you should ask
When testing mocks or prototypes, ask:
1. If this appeared once per day in your browser, would you keep it on?
2. Was there too much text?
3. Which headline did you care about most?
4. Which headline felt irrelevant?
5. Did the source names make you trust it more?
6. Would you want 3 headlines, 5, or 10?
7. Did this feel less annoying than email?
8. Would you rather see this morning, noon, or evening?
9. Does “from top tech voices” feel valuable enough by itself?
10. What would make this indispensable?
You want evidence for:
* retention
* trust
* low annoyance
* perceived time saved

17. Metrics that matter for MVP
Do not obsess over vanity metrics first.
Track:
Product quality metrics
* percentage of shown stories users say were actually relevant
* percentage of duplicates/noise
* average headline open/dismiss rate
* daily active users retaining after 7 days
Behavioral success metrics
* “I learned something useful today”
* “I didn’t have to open X”
* “This took <20 seconds”
* “I would keep this installed”
Red flag metrics
* muted quickly
* ignored entirely
* summaries feel obvious
* users say “I already saw all this”
* users say “this is just Twitter but worse”

2. Bad ranking
